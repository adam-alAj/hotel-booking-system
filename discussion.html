📁 Root Files (src/)
Entry points and utility files that initialize the React application, define global routing, and provide shared helper functions.
📄 App.js Root component — routing, layouts, role-based guards ▶
Purpose
The root component that defines the entire application routing structure, layout composition, and role-based access control. It is the single entry point that orchestrates which pages and layouts render based on authentication state and user roles.
Internal Details
  • Uses react-router-dom v6 with BrowserRouter, Routes, Route, Navigate, and Outlet
  • Defines two layout wrappers: CustomerLayout (wraps with CustomerNavbar + CustomerFooter) and ManagerLayout (wraps with ManagerNavbar + ManagerFooter)
  • Implements three route guards: CustomerGuard, ManagerGuard, and PublicRoute
  • Helper function checkIsManager(user) inspects user.roles array for 'HOTEL_MANAGER' or 'ADMIN'
  • All guards show a loading spinner while AuthContext resolves the user session
  • Routes: Public (/login, /signup) · Customer (/, /hotel/:id, /bookings, /search, /ai) · Manager (/manager-hotels, /manager-dashboard/:hotelId) · Catch-all → /login
System Role
This is the architectural backbone. Every page in the app is rendered through this file's routing tree. The guard pattern ensures security at the frontend level, complementing backend JWT validation.
Discussion Q&A
Q Why use route guards instead of checking roles inside each page component? ▶
✅ Answer
Centralizing access control in guards follows the Single Responsibility Principle. Pages focus on their UI logic, while guards handle authorization. This prevents code duplication and ensures no page accidentally renders without proper auth checks.
Q What happens if a manager tries to access a customer route like /bookings? ▶
✅ Answer
The CustomerGuard checks checkIsManager(user) — if true, it returns <Navigate to="/manager-hotels" replace />, redirecting the manager to their portal. The reverse applies for customers trying to access manager routes.
Q Why wrap the entire app in <AuthProvider>? ▶
✅ Answer
The AuthProvider uses React Context to make authentication state (user, isAuthenticated, loading) available to every component in the tree without prop drilling. Guards, navbars, and pages all consume this context.
📄 index.js React entry point — mounts app into the DOM ▶
Purpose
The JavaScript entry point that mounts the React application into the DOM.
Internal Details
  • Uses ReactDOM.createRoot() — React 18 concurrent mode API
  • Wraps <App /> in <React.StrictMode> for development warnings
  • Targets document.getElementById('root') in public/index.html
System Role
Standard React bootstrapping file. No custom logic — its role is purely to initialize the React tree.
Discussion Q&A
Q What does React.StrictMode do? ▶
✅ Answer
It activates additional development-only checks: detecting unsafe lifecycle methods, warning about legacy APIs, and double-invoking certain functions to detect side effects. It has zero impact in production builds.
📄 utils/imageUrl.js Resolves relative image paths to full backend URLs ▶
Purpose
A utility function that resolves image URLs by prepending the backend base URL to relative paths stored in the database.
Internal Details
  • Exports getImageUrl(url) function
  • Logic: if url is falsy → returns empty string; if starts with 'http' → returns as-is; otherwise → prepends http://localhost:8080
  • Solves mismatch where DB stores relative paths like /images/hotels/h1.avif but React dev server runs on a different port
System Role
Used by HotelCard, HotelDetail, ManagerHotels, and ManagerRoomTypesTab to correctly resolve image sources. Without this utility, all hotel images would 404.
Discussion Q&A
Q Why not use a proxy configuration in the React dev server instead? ▶
✅ Answer
A proxy can be unreliable for static assets. The explicit URL resolution is more transparent, works in all environments, and makes the dependency on the backend URL visible in code.
Q What would you change for production deployment? ▶
✅ Answer
Replace the hardcoded http://localhost:8080 with an environment variable (process.env.REACT_APP_API_URL) so the same build works across dev, staging, and production environments.
📁context/
Provides global application state using React Context API — specifically authentication state that needs to be accessible from any component in the tree.
📄AuthContext.js Manages login, logout, registration, token refresh, and session persistence ▶
Purpose
Manages the entire authentication lifecycle: login, registration, logout, session persistence, and token refresh event handling. Exposes auth state and actions to the entire component tree via React Context.
Internal Details
  • Creates AuthContext with createContext(null)
  • State: user (current user object), loading (true while checking stored token), isAuthenticated (boolean)
  • On mount, calls checkAuth(): reads accessToken from localStorage, calls authAPI.getCurrentUser() to validate
  • Listens for custom 'auth:logout' DOM event (dispatched by API interceptors on refresh failure)
  • Exposes: login(email, password), register(userData), logout()
  • Custom hook useAuth() provides type-safe context consumption
System Role
Every protected route guard consumes this context. The Navbar uses it for user display. Login/Signup pages call its methods. This is the single source of truth for "who is the current user?"
Discussion Q&A
QWhy store tokens in localStorage instead of httpOnly cookies?▶
✅ Answer
localStorage allows the SPA to attach the JWT in the Authorization header for API calls. httpOnly cookies require CSRF token handling. For a university project with separate frontend/backend, localStorage + Bearer tokens is the standard SPA pattern. In production, httpOnly cookies with CSRF protection would be more secure against XSS.
QWhat happens when the access token expires mid-session?▶
✅ Answer
The API interceptor catches 401 responses, uses the stored refresh token to obtain a new access token, retries the failed request, and updates localStorage. If the refresh also fails, it dispatches the 'auth:logout' event which AuthContext listens for, clearing the session.
QWhy use a custom event (auth:logout) instead of calling logout directly?▶
✅ Answer
The API interceptor lives outside the React component tree — it cannot call React hooks or context methods directly. The custom DOM event bridges this gap: the interceptor dispatches it, and the AuthProvider's useEffect listener handles the state cleanup.
📁services/
Contains all HTTP communication logic. Each file creates an Axios instance with interceptors for JWT attachment and automatic token refresh. Endpoints are organized by user role.
📄api.jsBase Axios instance + auth API + token refresh interceptor▶
Purpose
The base API service used for authentication-related endpoints. Creates the foundational Axios instance with request/response interceptors that handle JWT token management.
Internal Details
  • Base URL: http://localhost:8080
  • Request interceptor: attaches Bearer {token} from localStorage to every request
  • Response interceptor: on 401 → checks if refreshing → calls /api/auth/refresh → updates tokens → retries original request
  • Subscriber queue: prevents multiple simultaneous refresh calls via refreshSubscribers[]
  • Exports: login(), register(), logout(), getCurrentUser(), refreshToken()
Discussion Q&A
QWhy have three separate Axios instances instead of one shared instance?▶
✅ Answer
Separation of concerns — each instance can have different base configurations if needed (e.g., different timeouts, headers for multipart uploads). It also makes the codebase more modular: customer APIs, manager APIs, and auth APIs are independently maintainable.
QExplain the subscriber pattern in the refresh interceptor.▶
✅ Answer
When multiple API calls fail simultaneously with 401, only the first triggers a refresh. Subsequent calls are queued in refreshSubscribers[]. Once the refresh completes, onTokenRefreshed() iterates the queue providing the new token so each queued request can retry. This prevents multiple simultaneous refresh calls.
📄customerApi.jsHotels, rooms, bookings, payments, feedback endpoints▶
Purpose
Provides all API endpoints accessible to customer-role users: hotels, room types, bookings, payments, and feedback.
Internal Details
  • hotelAPI: getAllHotels(page, size), getHotelById(id), searchHotels(searchRequest)
  • roomTypeAPI: getRoomTypesByHotel(hotelId), getRoomTypeById(id)
  • bookingAPI: createBooking(data), cancelBooking(id), getMyHistory(), requestExtension(), confirmExtensionPayment()
  • paymentAPI: createPayment(paymentData)
  • feedbackAPI: createFeedback(data), getAllFeedbacks(page, size)
Discussion Q&A
QWhy does searchHotels use POST instead of GET?▶
✅ Answer
The search request contains a complex object with multiple optional filters (name, city, minRating, maxRating, pagination, sorting). Using POST with a JSON body is cleaner than encoding all these as query parameters, especially when values can be null. The backend expects a @RequestBody DTO.
QHow does the booking flow work end-to-end?▶
✅ Answer
1) User selects dates in BookingModal → 2) createBooking() sends hotel/room/dates to backend → 3) Backend calculates price with dynamic pricing rules and returns booking with quotedPrice → 4) User proceeds to PaymentModal → 5) createPayment() confirms the booking. The price is calculated server-side to prevent manipulation.
📄apiManager.jsHotel CRUD, rooms, room types, pricing rules, base prices▶
Purpose
Provides all API endpoints for hotel manager operations: hotel CRUD, room management, room types, bookings overview, base prices, and pricing rules.
Internal Details
  • Hotels: getMyHotels(), createHotel(formData), updateHotel(id, formData), deleteHotel(id)
  • Rooms: getHotelRooms(), createRoom(), patchRoom(), deleteRoom()
  • Room Types: getHotelRoomTypes(), createRoomType(formData), patchRoomType(), deleteRoomType()
  • Pricing Rules: createPricingRule(), patchPricingRule(), deletePricingRule(), assignRuleToRoomTypes()
  • Base Prices: setBasePrice(), patchBasePrice(), deleteBasePrice()
  • Hotel create/update use multipart/form-data for image upload
Discussion Q&A
QWhy use multipart/form-data for hotel creation?▶
✅ Answer
Hotels include an image file upload alongside text fields. multipart/form-data is the standard encoding for forms that include binary file data. The Axios instance overrides the default application/json content-type header for these specific calls.
QWhat's the difference between patchRoom and updateHotel?▶
✅ Answer
patchRoom uses HTTP PATCH (partial update — only send changed fields), while updateHotel uses HTTP PUT (full replacement). This follows REST conventions: PATCH for partial modifications, PUT for complete resource replacement.

The main landing page that displays featured hotels in a paginated grid, a hero section with platform statistics, feature highlights, an advertising section, and customer feedback.

Discussion Q&A
QWhy store searchParams in localStorage?▶
✅ Answer
When navigating from Home to HotelDetail and opening the BookingModal, the modal pre-fills check-in/check-out dates from these stored params. localStorage persists across page navigations without needing to pass props through the route.
QHow does pagination work with the backend?▶
✅ Answer
The backend returns a Spring Data Page object with content[], totalPages, pageNumber. The frontend tracks page state, passes it to the API call, and renders navigation controls. Each page change triggers a new API call — this is server-side pagination, not client-side.

Displays comprehensive information about a single hotel: hero image, rating, room types with amenities and pricing, and a booking flow.
Internal Details

Discussion Q&A
QWhy fetch hotel and room types separately?▶
✅ Answer
The backend follows RESTful resource separation — hotels and room types are different entities with different endpoints. Fetching them in parallel is still fast, and it allows room types to be reused independently (e.g., in manager views).
QHow is the price displayed before booking?▶
✅ Answer
The room type card shows the basePrice from the room type entity. The actual booking price (which includes dynamic pricing rules) is calculated server-side only when the booking is created. The BookingModal shows the final quotedPrice returned by the backend.
📄MyBookings.jsxBooking history — cancel, extend, pay actions▶
Purpose
Displays the authenticated customer's booking history with status indicators, and provides actions: cancel, extend, and pay.
Internal Details

Discussion Q&A
QWhat's the booking lifecycle?▶
✅ Answer
PENDING (created, awaiting payment) → CONFIRMED (paid) → COMPLETED (after checkout date) or CANCELLED (user cancelled). Extensions create a sub-entity linked to the booking. EXPIRED occurs if payment isn't made within a time window.

📄Login.jsAuth page — email/password form, role-based redirect▶

QWhy use window.location.href instead of navigate() for post-login redirect?▶
✅ Answer
A full page reload ensures the AuthProvider re-initializes with the new token, all components re-mount with fresh auth state, and any stale cached data is cleared. Using navigate() might leave some components in a pre-auth state.
📄Signup.jsRegistration page — form validation, password strength, role selection▶
Internal Details

Discussion Q&A
QIs client-side password strength validation sufficient?▶
✅ Answer
No — it's a UX enhancement only. The backend should enforce its own password policy (and does, via Spring Security's BCryptPasswordEncoder). Client-side validation provides immediate feedback but is never a security boundary.

QWhy implement search as a modal instead of a separate page?▶

A modal keeps the user in context — they don't lose their place on the current page. It's faster (no page load), feels more responsive, and follows the pattern of modern search UIs (Spotlight, Command+K). The actual results still navigate to a dedicated page.

A useEffect hook adds a keydown event listener to document. When showSearch is true and Escape is pressed, it calls closeSearch(). The listener is cleaned up on unmount to prevent memory leaks.
📄HotelCard.jsxHotel listing card — stars, badges, wishlist, CTA▶

QHow does the half-star rating work?▶
✅ Answer
Rating (0-10) is divided by 2 to get a 0-5 scale. Math.floor() gives full stars, and checking % 1 >= 0.5 determines if a half star is needed. Half stars use an SVG linearGradient with 50% gold / 50% transparent to visually split the star.
📄BookingModal.jsxTwo-phase booking creation — date selection → price → payment▶
Internal Details
  • Two-phase flow: Date selection → createBooking() → backend returns quotedPrice → "Continue to Payment" → renders PaymentModal
  • Date inputs: onKeyDown prevents manual typing; onClick opens native picker
  • Min date constraints: check-in ≥ today, check-out ≥ check-in + 1 day

Discussion Q&A
QWhy not let the user see the final price before creating the booking?▶
✅ Answer
The backend applies complex dynamic pricing rules. Calculating this requires creating a booking entity server-side. The two-phase approach (create → show price → pay) ensures price accuracy while giving the user a chance to review before paying.
📄PaymentModal.jsxPayment processing — Credit Card, PayPal, Bank Transfer▶
Internal Details
  • Three payment methods: CREDIT_CARD, PAYPAL, BANK_TRANSFER
  • Card number: auto-formats with spaces every 4 digits; CVV max 3 digits; expiry auto-inserts /
  • isFormValid computed via useMemo — enables/disables submit button
  • Field-level error display on blur via touched state tracking
  • On success: animated checkmark → redirects to /bookings after 2s
  • Retry logic: on 500 error, refreshes booking data and retries once
Discussion Q&A
QIs the credit card data sent to your backend?▶
✅ Answer
No — only the bookingId and method are sent to the backend. The card form is a UI simulation for the university project. In production, you'd integrate with a payment gateway (Stripe, PayPal SDK) that tokenizes card data client-side, never exposing raw card numbers to your server.
QWhy validate expiry date against the current date?▶
✅ Answer
An expired card will always be declined. Catching this client-side saves a round-trip to the server and provides immediate feedback. The comparison uses new Date(2000 + yy, mm) > new Date() — month is 1-indexed in the input but the Date constructor treats it as 0-indexed.

✅ Answer
The backend calculates it based on the same dynamic pricing rules used for the original booking, applied to the additional nights. The frontend only displays the result — it never calculates prices locally.
📄FeedbackSection.jsxCustomer reviews carousel — submit form, star ratings, avatars▶

QHow are avatar colors generated deterministically?▶
✅ Answer
A hash is computed from the user's name by summing character codes with bit shifting (charCodeAt(i) + ((hash << 5) - hash)). The hash modulo the gradient array length selects a consistent gradient. Same name always gets the same color.

📁components/ai/

📄AiChatWidget.jsxFloating AI chat — session management, typing indicator, minimize states▶
Internal Details
  • State: isOpen, isMinimized, input, messages[], isLoading, sessionId
  • Session ID: session-${random} generated once per mount to maintain conversation context
  • Auto-scrolls to bottom on new messages via useRef + scrollIntoView
  • Toggle states: closed (FAB button only) → open (full chat) → minimized (header only)
  • Uses lucide-react icons and framer-motion for animations
Discussion Q&A
QWhy use a separate session ID instead of the user's auth token for conversation tracking?▶
✅ Answer
The session ID tracks a single conversation thread. A user might have multiple conversations (different tabs) or the AI service might not need to know the user's identity for basic Q&A. The token is sent separately for authenticated features.
QWhy is the AI service on a different port (8082)?▶
✅ Answer
It's a separate microservice. Separating it from the main booking API (8080) follows microservice architecture principles: independent scaling, deployment, and failure isolation. The AI subsystem has its own dependencies (ML models, vector databases, etc.).
📄aiService.jsHTTP layer for AI microservice — uses fetch() not Axios▶
Internal Details
  • Base URL: http://localhost:8082/api/ai
  • Single exported function: sendChatMessage(message, sessionId, token)
  • Uses native fetch() (not Axios) — keeps AI service independent from main API layer
  • Sends POST to /chat with JSON body: { message, sessionId }
  • Attaches Authorization: Bearer {token} header if token is provided
Discussion Q&A
QWhy use fetch() here instead of Axios like the other services?▶
✅ Answer
The AI service is architecturally separate from the main booking API. Using fetch() avoids coupling it to the Axios interceptor chain (which handles booking API token refresh). The AI service has simpler auth needs — if the token is invalid, the chat just shows an error rather than attempting refresh.

QHow does image preview work before upload?▶

When the user selects a file, a FileReader reads it as a data URL (readAsDataURL). This base64-encoded string is set as the src of an <img> tag, showing the preview instantly without any server round-trip.
📄ManagerDashboard.jsxTabbed dashboard — rooms, room types, bookings, pricing▶
Internal Details
  • Extracts hotelId from URL params
  • Fetches ALL data in parallel using Promise.all() — 6 concurrent API calls
  • Each fetch has .catch(() => ({ data: [] })) — one failed fetch doesn't break the entire page
  • Tab navigation: Rooms, Room Types, Bookings, Pricing Rules, Base Prices
  • onRefresh callback passed to child tabs — allows child to trigger data re-fetch after mutations
Discussion Q&A
QWhy fetch all data upfront instead of lazily per tab?▶
✅ Answer
Hotel management data is relatively small. Fetching everything upfront eliminates loading delays when switching tabs, providing a snappier UX. The parallel fetch with Promise.all keeps the initial load fast.
QWhat's the onRefresh pattern?▶
✅ Answer
Each tab receives an onRefresh callback that re-fetches all dashboard data. When a tab creates/updates/deletes an entity, it calls onRefresh() to ensure all tabs have consistent data (e.g., creating a room type affects both the Room Types tab and the Rooms tab's dropdown).
📄ManagerRoomsTab.jsPhysical room CRUD — number, room type, status▶

QWhat's the relationship between rooms and room types?▶
✅ Answer
A room type defines the category (Deluxe, Suite) with amenities and base price. A room is a physical instance — "Room 301" of type "Deluxe". Multiple rooms share one room type. When a customer books a "Deluxe" room, the system assigns an available room of that type.

QHow do base prices interact with pricing rules?▶
✅ Answer
The pricing engine first looks up the applicable base price for the room type and date range. Then it applies all active pricing rules as percentage adjustments. Final quoted price: basePrice × (1 + sum of adjustments).

Discussion Q&A
QHow does rule priority work?▶
✅ Answer
When multiple rules apply to the same booking, priority determines the order of application. Lower priority numbers are applied first. This matters because percentage adjustments compound — a -10% early-bird followed by +5% weekend gives a different result than the reverse.
QWhat's the difference between EARLY_BIRD and LAST_MINUTE?▶
✅ Answer
EARLY_BIRD applies when booking is made many days before check-in (minDaysBeforeCheckin is high) — typically a discount to encourage advance booking. LAST_MINUTE applies when booking is close to check-in (maxDaysBeforeCheckin is low) — can be a discount to fill empty rooms or surcharge for urgency.

🧠 Key Technical Concepts
⚛️ React Component Lifecycle

The application uses functional components exclusively with React Hooks for lifecycle management:

HookUsage in this project
useStateLocal component state (forms, loading flags, modal visibility)
useEffectSide effects (API calls on mount, event listeners, cleanup)
useRefDOM references (auto-scroll in chat, input focus)
useMemoMemoized computations (form validation in PaymentModal)
useCallbackStable function references (modal close handlers)
useContextConsuming AuthContext without prop drilling

No class components are used. The useEffect cleanup pattern is consistently applied for event listeners and scroll locks.

🗄️ State Management Strategy

The project uses a lightweight, context-based approach rather than Redux or Zustand:

LayerToolScope
Global auth stateReact Context (AuthContext)Entire app
Page-level datauseState in page componentsSingle page
Cross-page persistencelocalStorageSearch params, tokens, theme
Component-local UIuseState in componentsSingle component

Why no Redux? The app has a single global concern (authentication). All other state is page-scoped — hotels on the Home page don't need to be shared with the Bookings page. This keeps the architecture simple and avoids boilerplate.

🔀 Routing & Navigation

Built with React Router v6:

  • Nested Routes with Outlet: CustomerLayout and ManagerLayout use <Outlet /> to render child routes within their layout wrapper
  • Route Guards: CustomerGuard, ManagerGuard, PublicRoute — check auth state before rendering
  • Programmatic Navigation: useNavigate() for post-action redirects (after login, after search)
  • URL Parameters: useParams() for dynamic routes (/hotel/:id, /manager-dashboard/:hotelId)
  • Query Parameters: useLocation() + URLSearchParams for search filters
  • Catch-all Route: <Route path="*"> redirects unknown paths to login
🌐 API Communication

All HTTP communication uses Axios with a consistent interceptor pattern:

Request Flow: Component → API Service → Axios Instance → Request Interceptor (attach JWT) → Backend Response Flow: Backend → Response Interceptor (check 401 → refresh token → retry) → API Service → Component
  • Three separate Axios instances: api (auth), customerApi (customer), apiManager (manager)
  • Automatic token refresh: 401 responses trigger refresh flow transparently
  • Subscriber queue: Prevents multiple simultaneous refresh calls when parallel requests fail
  • Multipart uploads: Hotel/room-type image uploads use FormData with overridden content-type
🔐 Authentication & Authorization Flow
AUTH FLOW: 1. User submits credentials (Login/Signup) 2. Backend validates → returns {accessToken, refreshToken} 3. Tokens stored in localStorage 4. AuthContext fetches user profile (GET /api/users/me) 5. User object (with roles[]) stored in context state 6. Route guards read context → allow/redirect TOKEN REFRESH: 1. API call returns 401 (token expired) 2. Interceptor sends refresh token to /api/auth/refresh 3. Backend validates → returns new token pair 4. New tokens saved to localStorage 5. Original failed request retried with new access token 6. If refresh fails → clear tokens → dispatch logout event ROLES: CUSTOMER → CustomerGuard routes HOTEL_MANAGER → ManagerGuard routes ADMIN → ManagerGuard routes (same access)

⚠️ Important: Frontend guards are a UX convenience. The real security boundary is the backend — even if a user bypasses frontend guards, API calls will fail with 403 Forbidden if their JWT doesn't have the required role.
